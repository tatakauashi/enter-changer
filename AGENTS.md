# AGENTS.md

## このファイルの目的

このリポジトリは `enter-changer` という Chrome 拡張機能です。

この拡張の目的は、AI チャット系の Web 画面において入力時のキー動作を次のように変更することです。

- `Enter` で改行
- `Ctrl + Enter` で送信

初期の対応対象は次のサービスです。

- ChatGPT
- Claude
- Gemini
- Grok

また、次の設定機能を持つことを前提とします。

- 機能全体の ON / OFF
- サイトごとの ON / OFF

---

## Codex への出力ルール

Codex は、このリポジトリに対して作業する際、**出力する内容を原則としてすべて日本語にすること**。

対象は次を含みます。

- チャットでの説明
- 提案文
- 実装方針の説明
- README などのドキュメント文
- コード中のコメント
- 必要に応じて作成する補助文書
- ユーザーが記載を指示した仕様・コメント・その他の文章

特別な理由がない限り、英語を主文にしないこと。  
ファイル名や API 名、ブラウザ仕様上の識別子、プログラミング言語の構文そのものは必要に応じて英語のままでよいですが、**人間向けの文章は日本語で記述すること**。

## ユーザーとの対話ルール

- **実装前に必ず質問・提案があれば行う** 。
- ユーザーが回答しやすいよう、質問は番号付き列挙で聞く。

## プロジェクトの優先順位

実装判断では、次の順で優先してください。

1. **日本語 IME 入力を壊さないこと**
2. **小さく、理解しやすく、保守しやすい実装にすること**
3. **まず ChatGPT 対応を安定させること**
4. **サイト固有の処理を共通処理から分離すること**
5. **不要な依存関係や複雑なビルド構成を持ち込まないこと**

このプロジェクトは、まずは **素の JavaScript による小さな Chrome 拡張** として始めます。  
明確な必要がない限り、大きなフレームワークは導入しないでください。

---

## 期待する動作

有効化された対象サイトでは、次の動作を期待します。

- `Enter` で送信せず、改行する
- `Ctrl + Enter` で送信する
- キー操作の変更は、実際のチャット入力欄に対してのみ適用する
- 通常のブラウザショートカットを邪魔しない
- 日本語 IME 変換中の Enter を邪魔しない

対象サイトが無効になっている場合、または拡張機能全体が無効になっている場合は、次の状態にしてください。

- 元のページの挙動をそのまま維持する
- キー動作を強制的に変更しない

## 初期段階ではやらないこと

明示的に依頼されない限り、初期段階では次のものを導入しないでください。

- React

- 分析基盤
- クラウド連携

まずの目標は、**動くことがわかりやすい最小構成の拡張** です。

## 想定するディレクトリ構成

強い理由がない限り、次のような構成を基本にしてください。

```text
enter-changer/
├─ manifest.json
├─ src/
│  ├─ background.js
│  ├─ content.js
│  ├─ popup.html
│  ├─ popup.js
│  ├─ styles.css
│  └─ sites.js
├─ icons/
│  ├─ icon16.png
│  ├─ icon48.png
│  └─ icon128.png
└─ README.md
```

### 各ファイルの役割

* `manifest.json`

  * Chrome 拡張の定義を記述する
  * 権限、popup、background、content script などを定義する

* `src/content.js`

  * 対象サイトに注入されるメイン処理
  * キー入力監視
  * Enter / Ctrl+Enter の制御
  * 共通処理とサイト固有処理の呼び分け

* `src/sites.js`

  * サイトごとの差分を集約する場所
  * URL 判定
  * 入力欄判定
  * 送信処理
  * サイトごとのセレクタ
  * サイト固有の注意点

* `src/popup.html` / `src/popup.js`

  * 拡張アイコンから開く設定 UI
  * 全体 ON / OFF
  * サイト別 ON / OFF
  * Chrome storage への保存と読込

* `src/background.js`

  * 必要最小限の background 処理
  * 既定値の初期化などが必要であれば担当する
  * 重くしないこと

## 設計方針

### 1. サイト固有ロジックを分離すること

ChatGPT / Claude / Gemini / Grok 固有のセレクタや挙動を、コードのあちこちに散らさないでください。

サイト固有の処理は、原則として `src/sites.js` に集約してください。

例:

* ホスト名によるサイト判定
* 入力欄の見つけ方
* 送信ボタンの探し方
* サイト固有の送信手順
* サイトごとの注意点

### 2. 小さなアダプタとして考えること

各サイトについて、次の要素を持つ小さなアダプタとして設計してください。

* サイトの判定方法
* 対象入力欄の判定方法
* 送信方法
* 特記事項

### 3. 共通処理は安全で単純にすること

共通処理は、小さく、読みやすく、責務が明確な形にしてください。
早い段階で抽象化しすぎないこと。

### 4. DOM 前提を増やしすぎないこと

対象サイトの DOM は将来的に変わる可能性があります。

そのため、壊れやすいセレクタへの過度な依存を避けてください。
セレクタが必要な場合は、次を優先してください。

* アクセシビリティ属性
* 安定した role
* 意味が安定しているボタン
* 複数候補によるフォールバック

## 入力処理のルール

### IME 安全性は必須

日本語やその他 IME の変換中の Enter を邪魔してはいけません。

キー動作を書き換える前に、必ず変換中かどうかを確認してください。

最低限、次を考慮してください。

* `event.isComposing`
* Enter が変換確定に使われるケース
* IME 利用時の境界的な挙動

IME 安全性に不安がある場合は、**処理しない方を優先** してください。

### 対象入力欄だけに適用すること

ページ全体の Enter を横取りしてはいけません。

処理対象は、対象サイトのチャット入力欄として明確に判断できる要素だけにしてください。
たとえば次のようなものです。

* `textarea`
* `[contenteditable="true"]`
* `[role="textbox"]`

ただし、単にこれらに一致しただけではなく、**対象チャット UI に属すること** も確認してください。

### 既定のキー動作

有効な対象入力欄では、次を基本ルールにしてください。

* 修飾キーなしの `Enter` → 改行
* `Ctrl + Enter` → 送信

明示的な要望がない限り、他の送信ショートカットは追加しないでください。

特に次のキーは慎重に扱ってください。

* `Shift + Enter`
* `Meta + Enter`
* `Alt + Enter`

必要が明確でない限り、再定義しないこと。

---

## 送信処理のルール

送信処理は、次の優先順で実装してください。

1. 信頼できる送信ボタンが見つかるなら、そのクリックを優先する
2. ボタン押下で足りない場合のみ、サイト固有の代替手段を使う

過度に強引な synthetic event の多用は避けてください。

送信ボタンのセレクタを使う場合は、次を守ってください。

* サイトごとにまとまっていること
* 複数候補を持てること
* すぐ壊れそうなセレクタに依存しすぎないこと

## 設定データの方針

設定は Chrome 拡張の storage を用いて保存してください。

想定するデータ形は、まずは次のようなものです。

```json id="rlrj2u"
{
  "enabled": true,
  "sites": {
    "chatgpt": true,
    "claude": true,
    "gemini": true,
    "grok": true
  }
}
```

方針:

* 安全な既定値を持つこと
* 値が欠けていても壊れないこと
* 初期段階では複雑なスキーマを持たないこと
* 命名を一貫させること

## 開発の進め方

実装は小さな単位で進めてください。

推奨順序:

1. Chrome 拡張の最小構成を作る
2. popup で全体 ON / OFF を実装する
3. ChatGPT 対応を実装する
4. ChatGPT のサイト別 ON / OFF を実装する
5. Claude 対応を追加する
6. Gemini 対応を追加する
7. Grok 対応を追加する
8. README とコードの整理を行う

最初から全サイトを同時に完璧にしようとしないこと。
まず 1 サイトを安定させてから次へ進めてください。

## コード記述の方針

JavaScript は、平易で読みやすく書いてください。

優先すること:

* 関数を短く保つ
* 意味がわかる名前をつける
* 条件分岐を明示的に書く
* 制御フローを単純にする
* コメントは本当に価値があるときだけ書く

避けること:

* 過剰設計
* 深すぎる抽象化
* 読みにくい clever な書き方
* 小さな用途のためにライブラリを増やすこと

コード変更時の方針:

* 現在の構造を、明確な改善理由がない限り壊さない
* 差分は目的に集中させる
* 無関係なリファクタを混ぜない

## 手動確認チェックリスト

変更後は、可能な範囲で次を確認してください。

### 基本確認

* Chrome に拡張を読み込んだとき manifest エラーが出ない
* popup が開く
* 全体 ON / OFF が動く
* サイト別 ON / OFF が動く
* 設定が再読み込み後も保持される

### 入力確認

変更対象サイトごとに、次を確認してください。

* チャット入力欄を正しく判定できる
* `Enter` で改行できる
* `Ctrl + Enter` で送信できる
* 日本語 IME 変換が壊れない
* チャット入力欄以外には適用されない
* サイト無効時は元の挙動に戻る

### 退行確認

* ページが固まらない
* イベントが二重登録されない
* 通常の Enter で誤送信しない
* Ctrl + Enter で二重送信しない

## 安全性と変更方針

対象サイトは第三者の Web アプリであり、将来的に DOM が変わる可能性があります。

そのため、次の方針を守ってください。

* 保守的な実装を優先する
* 強引な DOM 書き換えは避ける
* 必要のない polling は避ける
* event delegation は慎重に使う
* フォールバックを単純に保つ

あるサイトの対応が難しくなった場合、共通処理を複雑化して全体を壊すより、問題をそのサイト固有処理に閉じ込めることを優先してください。

実装詳細に迷った場合は、次の方針を取ってください。

IME 安全性や保守性と衝突する要望がある場合は、**IME 安全性と保守性を優先** してください。
